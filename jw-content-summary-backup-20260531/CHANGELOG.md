## v3.8 (2026-05-17)

**中间阅读成本优化：脚本索引 + 动态路由 + 统一验证 + 软聚类**

目标：不改变最终 SUMMARY 效果，只减少中间 token 消耗。

**新增脚本：**
- `scripts/build_book_index.py`: deterministic 预处理；切章、抽首尾段/小标题、提取 definitions/cases/warnings/quotes snippets，生成 `preprocess/book-index.json`
- `scripts/cluster_candidates.py`: 候选软聚类，生成 `clusters.json`；只分组，不删除成员

**阶段 1 改造：**
- 从「读序言+TOC+章节首尾段」改为「读取 book-index.json + snippets」
- 输出重点从理解改为路由：章节角色表 + 动态 extractor 路由表

**阶段 2 改造：**
- extractor 从固定 5 个改为动态 2-5 个
- glossary 不再读全书，只读 `snippets/definitions.json` + 必要上下文
- V1 跨域验证移出 extractor，由主代理基于全文索引统一补证，避免定向切片误杀
- 软聚类后再进入阶段 3，减少重复候选阅读

**阶段 3 改造：**
- 快速导航从「每问 1-2 段」压缩为「每项一句话」
- cluster 数 >10 即触发展开策略确认：Top 10 完整七段 / 全部展开 / 按类型拆分
- 未完整展开的方法论进入附录，不丢失信息

**预估收益：**
- 在 v3.7 基础上再省 25-40% token
- 修复 v3.7 中 V1 在切片内验证导致的 false negative 风险

## v3.6 (2026-05-17)

**第二问新增「作者的独特洞见」子板块**

在第二问「这本书在谈什么」中，于「核心观点」与「最想让读者记住的判断」之间插入「作者的独特洞见」。

**独特洞见的三条判断标准：**
1. **反直觉** — 大多数人第一反应是「不对吧？」但读完论证后不得不服
2. **区别于常识** — 告诉没读过的人不会被说「这谁不知道？」
3. **可迁移** — 能用在完全不同的场景上

**每条独特洞见的格式：**
- 常识是什么（大多数人怎么想）
- 作者的颠覆（作者说了什么相反的）
- 为什么重要（不读就会犯什么错）
- 原文依据

**撰写自检**: 写完后回看——每条读起来像「说得对，有道理」还是「等等，这跟我一直以为的不一样」？前者只是优质复述，后者才是独特洞见。

**同步更新：**
- templates/SUMMARY.md.template: 第二问新增「作者的独特洞见」
- templates/INDEX.md.template: 摘要提示同步
- SKILL.md: 重要区分节描述更新，version→3.6
- CHANGELOG.md、README.md 同步

## v3.5 (2026-05-17)

**五问内容深化 + 引文上限调整**

**第一问「作者的写作意图」拆为四个子问题：**
1. 作者面对的时代问题是什么？（具体的问题情境，不是泛泛的时代标签）
2. 作者想纠正什么误解？（错误认知 + 为什么危害大到值得写书纠正）
3. 作者想说服读者接受什么观点？（读者读完后的观念转变，不是学到了什么方法）
4. 作者为什么要写？（综合以上三点的写作动机）

**第二问「这本书在谈什么」扩充：**
- 核心主旨（一段话提炼核心命题）
- 核心观点（3-5 个可辩论的主张，附原文引用）
- 作者最想让读者记住的判断（一句话）
- 论证方式（吸收原第三问的论证方式）

**第三问改为「全书的内容结构」（原第三问内容吸收进第二问）：**
- 按主题/逻辑模块拆分为 3-6 个大板块
- 禁止机械罗列「第一章讲了什么，第二章讲了什么」
- 每个板块写明核心命题 + 在整体论证中的角色

**第四、五问不变**

**引文上限调整**: 原文引用上限从 ≤150 字放宽至 ≤200 字

**同步更新：**
- templates/SUMMARY.md.template: 一至三问完全重写
- templates/INDEX.md.template: 摘要版同步
- methodology/04-stage4-summarize.md: 操作要点、引文上限、自检
- SKILL.md: 重要区分节、质量红线、自检清单、version→3.5

## v3.3 (2026-05-17)

**新增 P-Procedure 段（第七段）**

- 方法论单元从六段扩展为七段：R/I/A1/A2/E/P/B
- P 段定位于 E（执行要点）与 B（边界）之间
- P 段五个设计原则：
  1. 还原因果链 —— 每步写输入来源和输出流向，禁止无因果的编号列表
  2. 区分必经步骤（🔵）与条件步骤（🟡）—— 条件步骤显式写出触发条件
  3. 显式化隐性前提 —— 前置条件区块列出所需知识/信息/工具
  4. 区分步骤与原则 —— 禁用「保持客观」这类原则替代操作
  5. 反向验证 —— 写完自问：新手能否仅凭 P 段独立完成一次应用？

**同步更新：**
- SKILL.md: description、质量红线、自检清单、虚构作品边界说明 → 六段改七段
- methodology/04-stage4-summarize.md: 章节标题、P 段方法论完整章节、单元结构模板、质量门控
