# 研报深度分析（Research Report Analysis）

IRON LAW: 所有分析结论必须锚定研报原文证据。无原文引用 = 无结论。禁止编造分析师未表述的观点。

Red Flags（出现以下情况必须回退重检）：
- 语气判断无原文依据 → 回退到原文定位
- 隐含假设非报告实际依赖 → 重新扫描核心逻辑链
- 预期差判断无市场参照 → 先确认一致预期再判断偏离

## Step 0: 输入识别 ⛔ BLOCKING

识别输入类型：
- **单篇**：1份研报文本/文件 → 执行 Step 1 → 跳过 Step 2
- **多篇**：≥2份研报文本/文件 → 执行 Step 1（逐篇）→ 执行 Step 2

确认信息：
- 研报来源（机构、分析师、日期）
- 分析目标（个股/行业/宏观）
- 输出偏好：默认保存为 Markdown 文件到研报源目录；用户指定"JSON"时仅输出 JSON 结构化版

→ 进入 Step 1

## Step 1: 单篇研报提取（阶段1）

加载 `references/extraction-guide.md`，按以下维度逐篇提取。

### 1.1 核心观点

问自己：这篇报告最核心的一句话结论是什么？分析师给出什么评级？

- 核心观点：一句话总结
- 评级：买入/增持/中性/减持/卖出（仅个股/行业报告有此字段，策略/宏观报告留空）
- 语气倾向：强烈看多 / 审慎看多 / 中性 / 审慎看空 / 强烈看空
- 语气置信度：极度自信 / 较为确定 / 模棱两可 / 相当谨慎
- 语气证据：**必须引用至少一处原文**

### 1.2 隐含假设

问自己：如果什么条件不成立，这篇报告的核心结论就会崩溃？

逐条列出报告依赖的关键假设，评估：
- 依赖度：核心驱动 / 辅助条件 / 背景假设
- 脆弱性：高（单一变量可推翻）/ 中（需多个变量同时变化）/ 低（基本面稳固）
- 脆弱性依据：判断理由

⚠️ 只提取报告实际依赖的假设，不要推测分析师"可能"的假设。

### 1.3 预期差

问自己：市场当前的一致预期是什么？这篇报告是强化还是挑战了共识？

- 市场共识：推断当前市场一致预期
- 报告立场：强化共识 / 挑战共识 / 提出差异化视角
- 分歧详情：分歧点具体描述
- 影响推演：若分析师正确，预期差收益或风险有多大

⚠️ 无明确分歧时写"强化共识"，不强行制造预期差。

### 1.4 关键论据

采用"核心论点→分论点→论据"三层结构：

**核心论点**：支撑报告核心观点的主要逻辑线（通常2-4条）。
每个核心论点下挂载：
- 核心论点：论点内容
- 分论点列表：每条分论点包含——
  - 分论点：具体内容
  - 论据类型：数据 / 逻辑推演 / 政策引用 / 行业调研
  - 论据详情：具体数据或事实
  - 原文引用：直接引用研报原文（如有）

⚠️ 核心论点是对报告核心观点的逻辑支撑，分论点是对核心论点的进一步论证。每个分论点必须有独立的论据，禁止空洞的"观点套观点"。

### 1.5 催化剂

- 催化剂：事件/政策/数据
- 时间窗口
- 潜在影响：高/中/低

### 1.6 风险因素

- 风险：风险描述
- 可能性：高/中/低
- 影响描述：影响程度

### 1.7 验证方法

问自己：如何用可获取的数据验证分析师的假设？

- 待验证假设
- 验证路径
- 建议数据来源

### 1.8 操作建议

提取所有具体建议，无则留空：
- 目标价 / 买入价位 / 建议仓位 / 止损位 / 投资期限 / 具体策略

⚠️ 研报未给出的字段留空，禁止编造具体价位或仓位。

→ 单篇完成。若为多篇输入，对每篇重复 Step 1 后进入 Step 2。

### 批量研报处理策略（≥5篇）

当输入研报数量≥5篇时，采用分层处理：

1. **分层分类**：按重要性分 Tier 1（中期策略/年度策略等核心报告）和 Tier 2（周报/点评/快报等轻量报告）
2. **Tier 1 深度拆解**：执行 Step 1 全部8个维度，每个核心论点下挂载完整分论点+论据
3. **Tier 2 快速提取**：仅提取核心观点（一句话）、语气倾向、与 Tier 1 的一致性/矛盾点，不做完整8维度拆解
4. **跨报告对比聚焦 Tier 1**：Step 2 的一致性分析、矛盾识别、假设脆弱性热力图均基于 Tier 1 报告，Tier 2 仅作为补充证据
5. **输出分层**：Markdown 输出先放 Tier 1 深度拆解，再放 Tier 2 快速提取表格，最后放跨报告对比

分层判断标准：
- Tier 1 标题含"中期策略""年度策略""展望""深度报告"或篇幅>2000行
- Tier 2 标题含"周报""点评""快报""速评""全知道"或篇幅<1000行

⚠️ 不要对所有报告一视同仁地做深度拆解——14篇报告全做8维度会淹没核心信息。分层后读者能快速定位最重要的分析。

## Step 2: 多篇研报对比（阶段2）

加载 `references/comparison-guide.md`，基于 Step 1 的结构化结果进行深度对比。

### 2.1 一致性分析

问自己：哪些维度上分析师们达成了共识？共识的强度如何？

列出观点一致的维度，标注支持的报告编号。

### 2.2 矛盾识别

问自己：分析师们在哪些关键问题上产生了分歧？分歧的根源是什么？

对每个矛盾点：
- 分别陈述双方立场（带报告编号）
- 根源分析：假设不同 / 数据源不同 / 分析框架不同 / 时间窗口不同

### 2.3 假设脆弱性热力图

问自己：哪些假设被多篇报告同时依赖？这些假设有多脆弱？

找出≥2篇报告共同依赖的假设，评估：
- 依赖报告：依赖该假设的报告编号列表
- 脆弱性评分：共识依赖度 × 脆弱性的综合评分（高/中/低）
- 敏感性分析：若该假设不成立，对各报告结论的影响程度

⚠️ 这是系统性风险预警的核心输出。共同依赖+高脆弱 = 最大风险共鸣点。

### 2.4 预期差矩阵

问自己：哪些报告在提供"逆向思维"？逆向的程度有多大？

对每篇报告评估：
- 报告立场：强化共识 / 挑战共识
- 预期差大小：显著/中等/微小
- 争议程度：高/中/低

### 2.5 操作建议共识

汇总各报告的操作建议：
- 目标价共识：目标价区间
- 仓位建议共识
- 操作建议分歧：操作建议上的关键分歧

→ 进入 Step 3

## Step 3: 输出生成 ⚠️ REQUIRED

加载 `assets/output-template.json` 作为输出骨架。

### 单篇模式
输出完整 report_summaries[0] 对象。

### 多篇模式
输出完整 report_summaries[] + 比较分析。

### 输出格式与文件保存

**默认行为：将分析结果保存为 Markdown 文件。**

文件命名规则：
- 单篇：`研报分析_{机构}_{分析师}_{报告简称}_{YYYYMMDD}.md`
- 多篇：`研报对比分析_{目录名或主题}_{YYYYMMDD}.md`

保存位置：
- 优先保存到研报源文件所在目录（同级或子目录 `analysis/`）
- 若源目录无写入权限，保存到 `~/` 目录

文件内容结构（Markdown）：
1. 标题（报告名称 + 分析日期）
2. 一句话总结（核心观点）
3. 各报告深度拆解（按 Tier 1 → Tier 2 分层）
4. 跨报告对比（一致性/矛盾/假设脆弱性热力图/预期差矩阵）
5. 操作建议共识
6. 风险提示

同时在终端输出精简版摘要（核心观点 + 关键分歧 + 配置建议），引导用户阅读完整文件。

### 输出格式细节
- Markdown 版以"结论先行"结构：核心观点 → 隐含假设 → 预期差 → 完整提取
- JSON 版严格遵循 `assets/output-template.json` 字段规范（仅在用户明确要求时输出）
- 用户指定纯 JSON 时仅输出 JSON（不保存 Markdown）

## Step 4: 交付验证

### Pre-Delivery Checklist

#### 完整性
- [ ] 所有必填字段无遗漏（核心观点、隐含假设、预期差）
- [ ] 每篇研报均完成 Step 1 全部 8 个子维度
- [ ] 多篇模式：比较分析五个子模块均已输出

#### 证据链
- [ ] 语气证据包含至少一处原文引用
- [ ] 所有隐含假设均可在报告中找到逻辑支撑
- [ ] 预期差判断有市场参照基准
- [ ] 无任何编造的目标价、仓位或操作建议
- [ ] 关键论据的每个分论点均有独立论据支撑

#### 质量
- [ ] 假设脆弱性热力图标注了≥1个共同依赖假设（多篇模式）
- [ ] 矛盾识别覆盖了核心分歧点，非边缘细节
- [ ] JSON 输出可被下游系统解析（无语法错误）

## 反模式（Anti-Patterns）

- **编造预期差**：报告未挑战共识时，不要强行制造分歧 → 写"强化共识"
- **语气武断**：不要用主观感受判断语气 → 必须有原文证据
- **假设过度挖掘**：只提取报告实际依赖的假设，不是所有可能的前提
- **矛盾夸大**：措辞差异≠观点矛盾 → 只在核心逻辑不同时标注矛盾
- **操作建议补全**：分析师未给目标价时不要"推算" → 留空
- **一次性全量加载**：不要在 Step 0 就加载所有 references → 按步骤按需加载
- **论点无支撑**：分论点不能只有观点没有论据 → 每条分论点必须附论据类型和详情
