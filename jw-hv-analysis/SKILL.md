# 横纵分析法深度研究

> **方法论溯源**：横纵分析法由数字生命卡兹克（Khazix）提出，融合索绪尔历时-共时分析与竞争战略分析。核心：纵向追时间深度，横向追同期广度，交汇出判断。

### 术语定义

| 术语 | 定义 |
|------|------|
| 纵轴 / 纵向分析 | 沿时间轴追踪研究对象从诞生到当下的完整生命历程，以叙事故事呈现，不是年表 |
| 横轴 / 横向分析 | 在当前时间截面上，将研究对象与同赛道竞品/同类进行系统性对比 |
| 横纵交汇 | 纵向脉络与横向格局的交叉分析，产出前两步未明确说出的新判断（不是缩写版） |
| 独立来源 | 不同机构/个人/平台发布的信息。同一公司的官网和官方博客算1个来源；官网+第三方媒体算2个来源 |
| 一手来源 | 事件直接相关方发布的信息（官方公告、创始人发言、原始数据），非转载或解读 |
| 竞品 | 满足相同用户需求的产品/服务/方案。间接替代品也算（如"线下会议"是"视频会议软件"的间接竞品） |

## ⚠️ 写作规范（最先阅读）

**执行前必须熟记以下规则，违反任何一条即为不合格。**

### 绝对禁区（红线）

**AI味标记**：
- ❌ 套话："首先"、"其次"、"最后"、"综上所述"、"值得注意的是"、"不难发现"
- ❌ 空洞形容词："赋能"、"抓手"、"打造闭环"
- ❌ 教科书开头："在当今AI快速发展的时代"、"随着技术的不断进步"
- ❌ 高频踩雷词："说白了"、"意味着什么"、"这意味着"、"本质上"、"换句话说"、"不可否认"
- ❌ 空泛工具名：不说"AI工具"、"某个模型"，要说具体名字
- ❌ 编造场景：搜不到的信息诚实标注「该信息暂缺」，绝不编造

**报告禁区**：
- ❌ 年表流水账："2023年1月发布了A，2023年3月发布了B"
- ❌ 空洞概括："该公司在这一阶段实现了快速增长"
- ❌ 前面内容的缩写版充当"交汇洞察"

### 正确做法

- ✅ 用具体细节代替概括：「从2024年中期的1000万美元ARR到2025年底的10亿美元，增长曲线几乎是垂直的」

### 写作风格（三条核心约束）

1. **叙事驱动**：纵向部分要有故事弧线，有起承转合，不是冷冰冰的咨询报告。开头埋的钩子在结尾callback回来。
2. **先事实后判断**：每个观点必须有事实支撑，用「我觉得」「我的判断是」承认主观性。用"现象→表面解释→追问→核心洞察"层层剥开。
3. **交汇文化升维**：在横纵交汇部分，连接到更大的文化/哲学/历史参照物。

报告可以有聊天感（密度低于公众号），需要清晰章节结构（1-3万字没导航会迷路），不加公众号尾部。

核心原则：**质量优先**。每个关键节点都值得展开，不要为了压缩而跳过重要细节，也不要为了凑字数而注水。

## 第一步：联网信息收集

**核心原则**：信息质量决定报告质量。宁可多搜，不要浮于表面。

### 搜索关键词生成策略

根据研究对象类型，自动生成搜索关键词组合：

| 对象类型 | 关键词模板 | 示例 |
|---------|-----------|------|
| 产品 | "{产品名} history"、"{产品名} vs {竞品}"、"{产品名} funding" | "Notion history"、"Notion vs Confluence" |
| 公司 | "{公司名} founding story"、"{公司名} revenue"、"{公司名} layoffs" | "Stripe founding story" |
| 概念 | "{概念} origin"、"{概念} criticism"、"{概念} vs {类似概念}" | "Web3 origin"、"Web3 vs Web2" |
| 人物 | "{人名} career"、"{人名} interview"、"{人名} controversy" | "Sam Altman career" |

**多语言搜索**：先搜英文，再搜中文，再搜日文/韩文（如适用）。

### 搜索策略

1. **多轮搜索**：不要只搜一次就放弃。先用宽泛关键词发现线索，再用具体关键词深入。
2. **一手来源优先**：
   - 产品更新/技术决策 → 官方博客、GitHub Release Notes、创始人推文
   - 融资/商业数据 → 公司官方公告、SEC/工商文件
   - 用户口碑 → GitHub Issues、Reddit、Twitter/X、知乎
   - 学术/技术 → arXiv论文、Google Scholar
3. **交叉验证**：关键事实至少两个独立来源确认。单一来源标注为"待验证"。
4. **学术类必查 arXiv**：涉及技术范式、算法、AI模型时，调用 `curl -s "https://export.arxiv.org/api/query?search_query=all:关键词&max_results=10"` 获取论文。

### 工具使用

根据当前环境可用工具灵活选择：
- **WebSearch**：发现信息来源，获取摘要
- **WebFetch/Crawl**：已知URL时深入提取内容
- **GitHub API**：`curl -s "https://api.github.com/repos/{owner}/{repo}"` 获取仓库数据
- **本地文件**：研究对象在本地时，直接读取相关文件

### 信息充分性自检

搜索完成后，检查以下量化指标：

| 维度 | 最低标准 | 理想标准 |
|------|---------|---------|
| 信息源数量 | ≥5个独立来源 | ≥10个独立来源 |
| 一手来源占比 | ≥40% | ≥60% |
| 关键事实覆盖率 | 能讲完整故事，无明显断层 | 每个关键节点都有来源支撑 |
| 竞品信息 | 每个主要竞品≥2个来源 | 每个竞品≥4个来源 |
| 交叉验证 | 关键事实至少2个独立来源确认 | 关键事实至少3个来源确认 |

**自检清单**：
- [ ] 纵向：能讲完整故事吗？有明显信息断层吗？
- [ ] 横向：竞品列表完整吗？每个竞品信息够做对比吗？
- [ ] 来源：关键事实有可靠来源支撑吗？单一来源标注为"待验证"？

信息不够就补搜，不要凑合。

### ⚠️ Pitfall：扩写内容后禁区词反弹

扩写时容易重新引入禁区词。**防护流程**：扩写后先用 `search_files` 扫描禁区词，再跑质检脚本。详见 `references/forbidden-words-guide.md`。

## 第二步：纵向分析（Diachronic / Longitudinal）

沿时间轴，完整还原研究对象从诞生到现在的发展全貌。**这是报告的主体部分。**

### 内容要求

**起源追溯**：诞生背景？基于什么技术/理念/需求？创始团队是谁？他们之前做过什么？当时的行业环境？有没有关键事件直接促成诞生？

**诞生节点**：明确的首次发布/成立/提出时间，最初形态和定位，跟现在有什么不同。

**演进历程**：从诞生到现在，按时间顺序梳理所有关键节点——重大版本更新、融资事件、团队变动、战略转型、技术架构变化、用户规模里程碑、重大合作或收购、公关危机或争议事件。

**决策逻辑**：在每个关键节点，还原决策背后的原因。为什么选A不选B？当时约束条件是什么？哪些早期决策"锁定"了后来的发展方向？

**阶段划分**：把整个历程自然分为几个阶段（萌芽期、快速增长期、转型期等），每个阶段有核心特征和核心矛盾。

---

## 第三步：横向分析（Synchronic / Cross-sectional）

以当前时间点为切面，将研究对象与同赛道竞品/同类进行全面对比。

### 竞品场景判断

| 场景 | 处理方式 |
|------|---------|
| **A：无直接竞品** | 分析为什么没竞品（品类新/壁垒高/市场小）？未来最可能从哪冒出竞争者？间接替代方案？ |
| **B：少量竞品（1-2个）** | 逐一深入对比 |
| **C：竞品充分（3+）** | 选3-5个代表性竞品详细对比，其余简要提及 |

### 对比维度

根据研究对象类型灵活调整，至少覆盖：

**核心差异**：技术路线/核心方法论、产品形态/商业模式、目标用户/适用场景、核心优势与短板、定价/规模。

**用户视角**：真实用户口碑？社区评价中最多的优点和槽点？用户实际使用方式和官方定位有没有偏差？每个竞品「活成了什么样」？

**生态位分析**：在整个赛道版图中占据什么位置？当前格局是百花齐放、两强争霸、还是一家独大？

**趋势判断**：基于横向对比，机会和风险各是什么？

---

## 第四步：横纵交汇洞察

**报告精华段。** 把纵向脉络和横向格局结合起来，给出综合性新判断。**不是前面内容的缩写版。**

### 核心问题

1. **历史如何塑造了当下的竞争位置**：纵向历程中的哪些决策，决定了它今天在横向对比中的位置？
2. **竞品的纵向对比**：主要竞品的起源和演变路径有什么不同？如何导致了今天各自的特点？
3. **优势的历史根源**：今天的每个核心优势，能追溯到历史上的哪个节点或决策？
4. **劣势的历史根源**：今天的每个核心劣势，能追溯到哪个历史决策？当初的「好决策」有没有变成今天的包袱？
5. **未来推演**：三个剧本——最可能的、最危险的、最乐观的，每个要有逻辑支撑。

## 第五步：质检与交付

### 自动质检

交付前运行质检脚本（**注意：质检是最终确认手段，不是发现禁区词的主要手段——先手动扫描**）：

```bash
python3 scripts/quality_check.py "研究报告.md"
```

### 检查点保存（长报告必备）

每完成一步（纵向/横向/交汇），运行检查点脚本保存进度，防止中断丢失：

```bash
# 快捷模式：为研究对象生成所有阶段模板
python3 scripts/save_checkpoint.py --quick "研究对象名"

# 保存中间产物
python3 scripts/save_checkpoint.py --stage vertical --object "研究对象名" --data-file vertical.json

# 加载中间产物
python3 scripts/save_checkpoint.py --stage vertical --object "研究对象名" --load
```

### Anti-Patterns（执行层面）

- ❌ 只搜一次就写报告 → 多轮搜索，先宽后窄，信息不够就补搜
- ❌ 纵向写成年表流水账 → 必须是叙事故事体，有因果逻辑和决策还原
- ❌ 交汇写成前两步缩写 → 必须产出新判断、新视角
- ❌ 竞品只列名字不深入 → 每个竞品必须有用户口碑+生态位分析
- ❌ 搜不到就编造 → 诚实标注「该信息暂缺」
- ❌ 扩写后不重新扫描禁区词 → 每次扩写后必须重扫

脚本检查（子串匹配）：字数范围/禁区词汇/信息来源标注/章节结构完整性。

**质检结果语义**：
- **error（❌）**：必须修复后才能交付（禁区词、缺失章节）
- **warning（⚠️）**：不阻塞交付，但必须告知用户并让其决定是否修改（字数偏少/偏多、无来源标注）

**⚠️ Pitfall：禁区词汇反复出现（高频踩雷）**

质检脚本对禁区词汇做**子串匹配**——"最后一块拼图"会触发"最后"，"首先是一个"会触发"首先"。你在扩充内容时，极容易重新引入刚修掉的禁区词。

**正确流程（节省3-5轮返工）**：

1. **写完初稿后、首次质检前**：用 `search_files` 逐个扫描所有禁区词（"首先"、"最后"、"综上所述"、"值得注意的是"、"不难发现"、"赋能"、"抓手"、"打造闭环"、"这意味着"、"本质上"、"换句话说"、"不可否认"、"AI工具"、"说白了"），一次性全部修掉
2. **每次扩充内容后**：重复步骤1，不要信任自己刚写的内容"应该没问题"
3. **质检脚本只用来做最终确认**，不要把它当成发现禁区词的主要手段——手动扫描更快更全

**扩写时的禁区词速查表和防护流程**：详见 `references/forbidden-words-guide.md`

### 人工质检清单

**纵向维度（第二步）**：
- [ ] 纵轴是叙事故事体？有因果逻辑和时代脉络？不是年表流水账？
- [ ] 创始人/发起者背景和动机有足够深度？
- [ ] 每个关键节点都展开了，没有跳过重要细节？
- [ ] 决策逻辑有还原？不只是「发生了什么」，还有「为什么这么选」？

**横向维度（第三步）**：
- [ ] 横轴的竞品场景判断正确（A/B/C）？竞品分析够深？
- [ ] 用户口碑引用了真实用户声音？不只是官方宣传？

**交汇维度（第四步）**：
- [ ] 横纵交汇产出了新判断，不是前面内容的缩写版？
- [ ] 未来推演三个剧本都有逻辑支撑？

**写作风格（全文）**：
- [ ] 写作风格有节奏感、可读性？不是冷冰冰的咨询报告？
- [ ] 没有触犯绝对禁区？

**信息与来源（第一步）**：
- [ ] 关键事实标注了信息来源？
- [ ] 搜不到的信息标注了「暂缺」？

**交付（第五步）**：
- [ ] 交付物格式正确、结构完整、可读性好？
- [ ] 总字数在合理范围内（根据研究对象特征调整）？

### 交付路径

根据环境选择交付方式：

| 环境 | 方案 | 操作 |
|------|------|------|
| 飞书环境 | 飞书云文档（✅首选） | `cd /root/data && lark-cli docs +create --api-version v2 --doc-format markdown --content @文件名.md` |
| 其他 | 本地Markdown（✅推荐） | 保存为 `[研究对象名称]_横纵分析报告.md` |
| 需要PDF | PDF（⚠️备选） | `pip install weasyprint markdown --break-system-packages` 后运行 `python3 scripts/md_to_pdf.py input.md output.pdf`（安装极慢且可能失败） |

### 交付后确认

交付完成后，执行以下确认步骤：

1. **文件完整性检查**：
   - PDF/飞书文档是否可正常打开？
   - 报告结构是否完整（封面、目录、四章、来源）？

2. **链接有效性检查**（飞书/在线文档）：
   - 文档URL是否可访问？
   - 分享权限是否正确？

3. **交付确认清单**：
   - [ ] 文件已保存/上传到指定位置
   - [ ] 文件大小合理（PDF通常>100KB）
   - [ ] 报告标题和作者信息正确
   - [ ] 已告知用户交付位置和访问方式

报告末尾加上：
- **信息来源**：所有引用的来源清单，标注URL和访问时间
- **方法论说明**：简要说明横纵分析法来源（1-2句话）

### 报告结构模板（与五步流程的对应）

```
封面页（PDF/飞书自动生成）

目录

一、一句话定义              ← 第一步信息收集 + 第二步纵向分析的产物
[用一句话说清楚这个东西是什么]

二、纵向分析：从诞生到当下  ← 第二步：纵向分析
[完整的纵向叙事]

三、横向分析：竞争图谱      ← 第三步：横向分析
[横向对比分析]

四、横纵交汇洞察            ← 第四步：横纵交汇
[交叉分析和未来推演]

五、信息来源                ← 第一步信息收集的来源清单
[所有引用的来源列表]
```

## 不同研究对象类型的适配

核心原则不变（纵向追时间深度，横向追同期广度），侧重点不同：

**研究产品时**：纵轴关注版本迭代、技术路线演变、用户增长曲线、关键产品决策；横轴关注功能对比、性能对比、用户体验、定价。

**研究公司时**：纵轴关注创始团队、融资历程、战略转向、组织变革、关键人事变动；横轴关注商业模式差异、市场份额、营收对比、组织架构差异。

**研究概念时**：纵轴关注概念起源、如何流行起来、经历了哪些争论和演变；横轴关注与相近概念的区别、各自适用场景、不同阵营的论证。

**研究人物时**：纵轴关注个人经历、职业轨迹、关键决策、成长曲线、公开言论变化；横轴关注与同领域其他人物的对比。
