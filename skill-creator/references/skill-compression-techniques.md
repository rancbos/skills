# Skill Compression Techniques

> 来源：jw-company-analysis v3.29.0 SKILL.md 压缩（1010→974行，-36行）+ v4.1.6 压缩（513→497行，-16行）
> 当 SKILL.md 超过 1000 行软限制时，用以下技术压缩而不损失执行功能。

## 核心原则

**压缩的是人类参考引用，不是执行指令。** Agent执行必需的判断标准、评分公式、排除条件必须保留内联。压缩目标是📚人类参考（agent不读的理论背景）。

## 技术1：引用路径简写化

**问题**：SKILL.md 中有大量 `📚 详见 \`references/investment-theory.md\` §XX`，完整路径重复出现 80+ 次，每次浪费 ~40 字符。

**解决**：在 SKILL.md 概览区定义一次路径映射，后续用简写：

```markdown
**📖 引用简写**：📚§XX=investment-theory.md / 🧠=munger-mental-models / 💰=valuation-methods / 🌍=macro-context / 📊=zhang-xinmin / 🔧=scripts。agent只读内联内容，📚仅供人类查阅。
```

然后全文替换：
- `📚 详见 \`references/investment-theory.md\` §37` → `📚§37`
- `📚 详见 \`references/investment-theory.md\` §86, §89` → `📚§86,§89`

**节省**：每个引用节省 ~30 字符，80 个引用可节省大量行数（行折叠效应）。

**⚠️ 保留完整路径的情况**：
- 操作性引用（agent需要读的文件，如 checkpoint-schema.md）
- 文件末尾的总览引用
- 跨文件引用（非 investment-theory.md 的其他 reference 文件首次出现时）

## 技术2：同子步骤📚合并

**问题**：一个子步骤（如 Step 1.1）内有 6 个独立📚行，每个一行，占 6 行。

**解决**：合并为一行汇总引用：

```markdown
# 之前（6行）：
**特许经营权vs商品型企业**（巴菲特）：...📚§21
**经济特许权评估**（多尔西）：...📚§188
**林奇六类公司分类**：...📚§22,§83
**邱国鹭行业四维度**：...📚§23
**行业选择负面清单**（多尔西）：...📚§206
**冯柳三维**：...📚§24

# 之后（1行）：
**📚商业模式扩展**：特许经营权vs商品型(§21) / 经济特许权(§188) / 林奇分类(§22,§83) / 邱国鹭四维度(§23) / 负面清单(§206) / 冯柳三维(§24)
```

**哪些行可以合并**：
- 纯📚引用行（概念名+一句描述+§号，没有内联评分标准/公式/表格）
- 同一子步骤内的📚行

**哪些行不能合并**：
- 有内联定义的概念（如"林奇六类公司分类"的完整分类列表）
- 包含评分公式或判断标准的行
- 包含表格的行

**合并策略**：保留有内联定义的概念为独立行，将纯引用行合并为一行📚汇总。

## 技术3：删除独立"扩展概念"行

**问题**：每个子步骤末尾有 `> **扩展概念**：📚 详见 ... §XX, §YY` 行，这些§号已在上方内联内容中提及。

**解决**：直接删除。如果§号已在内联内容中出现，扩展概念行是冗余的。

**检测**：`grep -n '扩展概念\|扩展框架' SKILL.md` 找到所有扩展行，逐一检查§号是否已在上方出现。

## 技术4：引用简写说明精简

**问题**：引用分类说明占 4-8 行。

**解决**：压缩为 1 行：

```markdown
# 之前（8行）：
**📖 引用分类说明**：
- **SKILL.md内联**：agent执行必需的判断标准、评分公式、排除条件。agent直接使用。
- **📚人类参考**：标注`详见 references/...`的引用，指向详细理论背景（5000+行）。agent不需要读，人类深入研究时查阅。
- **🔧脚本工具**：pre_analysis.py、7track_boll.py等自动化脚本。agent调用执行。

# 之后（1行）：
**📖 引用简写**：📚§XX=investment-theory.md / 🧠=munger-mental-models / 💰=valuation-methods / 🌍=macro-context / 📊=zhang-xinmin / 🔧=scripts。agent只读内联内容，📚仅供人类查阅。
```

## 执行顺序

1. 先做引用路径简写化（最大收益，全文替换）
2. 再做同子步骤📚合并（逐子步骤处理）
3. 删除独立扩展概念行
4. 精简引用简写说明
5. 多行代码块内联化（公式类内容）
6. 相邻薄章节合并（<10行的相邻章节）
7. 相邻子步骤合并（连续的短子步骤）
8. 验证：`wc -l` + Step完整性 + 排除/降级/结论矩阵检查

## 验证清单

压缩后必须验证：
- [ ] 行数减少（`wc -l SKILL.md`）
- [ ] 所有 Step 标题完整（`grep '^## Step' SKILL.md`）
- [ ] 排除检查条件未丢失（`grep '❌排除' SKILL.md`）
- [ ] 降级检查条件未丢失（`grep '⚠️降级' SKILL.md`）
- [ ] 结论矩阵档位完整（`grep '⭐\|🟢\|🟡\|🔴' SKILL.md`）
- [ ] 评分公式完整（`grep '综合得分' SKILL.md`）
- [ ] 建仓终检清单完整（`grep '①' SKILL.md`）
- [ ] 版本号未被误改

## 典型收益

| 技术 | 典型节省 | 适用场景 |
|------|---------|---------|
| 路径简写化 | 20-40行 | >50处📚引用的大型skill |
| 同子步骤合并 | 15-30行 | 每个子步骤有3+📚行 |
| 删除扩展概念行 | 3-5行 | 有独立"扩展概念/框架"行 |
| 引用说明精简 | 3-7行 | 引用分类说明>3行 |
| 代码块内联化 | 4-10行 | 有2-5个多行公式代码块 |
| 相邻薄章节合并 | 4-12行 | 多个<10行的独立章节 |
| 相邻子步骤合并 | 3-8行 | 连续的短子步骤对 |
| **合计** | **50-110行** | 1000+行的大型skill |

## 技术5：多行代码块内联化

**问题**：独立的公式代码块（` ```...``` `）占 3-5 行，但公式本身可以内联到段落中。

**适用条件**：公式≤2行且无复杂格式（不需要对齐表格或分行注释）。

**示例**：
```markdown
# 之前（4行）：
**首选：DCF 所有者收益法**：
` ` `
内在价值 = Σ [所有者收益_t / (1+r)^t] + 终值
所有者收益 = 净利润 + 折旧摊销 - 维护性资本开支
` ` `

# 之后（1行）：
**首选：DCF 所有者收益法**：`内在价值 = Σ[所有者收益_t/(1+r)^t] + 终值`，所有者收益 = 净利润 + 折旧摊销 - 维护性资本开支。
```

**节省**：每个代码块 2-4 行。适用于安全边际公式、DCF公式等简单公式。

**⚠️ 不适用**：JSON schema 示例、多行 bash 脚本、需要对齐的表格化公式。

## 技术6：相邻薄章节合并

**问题**：两个相邻章节都<10行，各自独立占一个 `##` + 分隔线。

**适用条件**：两个章节在逻辑上相关（如"输出结构"和"能力边界声明"都是交付相关）。

**示例**：
```markdown
# 之前（11行，含分隔线）：
## 输出结构
详见 `templates/xxx.md`。文件命名：...
> ⚠️ **结论必须有论据和计算过程**...

---

## 能力边界声明
数据局限+模型局限+知识局限+时间局限+心理局限。**本分析不构成投资建议**。

# 之后（5行）：
## 输出结构与能力边界
详见 `templates/xxx.md`。文件命名：... ⚠️ 每个评分附带评分依据。
**能力边界**：数据局限+模型局限+知识局限+时间局限+心理局限。**本分析不构成投资建议**。
```

**节省**：每个合并 4-6 行（省去重复的章节标题+分隔线）。

## 技术7：相邻子步骤合并

**问题**：两个相邻子步骤（###）内容都很短（<5行），且逻辑上连续。

**适用条件**：子步骤 B 的输入是子步骤 A 的输出，或两者属于同一个执行动作的两个阶段。

**示例**：
```markdown
# 之前（10行）：
### 1.6 快速排除
1. **审计意见**：非标意见→❌排除
2. **商业模式可理解性**：一句话说不清→⚠️降级

### 1.7 信息完整性检查
确认已收集：基本信息、股价（≥3源）、3-5年财务...

# 之后（5行）：
### 1.6 快速排除与信息完整性
**排除条件**：①审计意见非标→❌排除 ②一句话说不清→⚠️降级。
**完整性检查**：确认已收集：基本信息、股价（≥3源）、3-5年财务...
```

**节省**：每个合并 3-5 行。适用于 Step 内的前置检查+数据确认等连续流程。
