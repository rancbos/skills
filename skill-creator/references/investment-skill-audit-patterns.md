# 投资分析类 Skill 审计常见问题模式

> 来源：jw-company-analysis v1.0→v1.1.3 审计修复经验（2026-06-01）+ v2.3→v2.4 张新民财报补充经验（2026-06-01）

## 常见问题清单

创建或审计投资分析类 skill 时，以下问题最容易出现：

### 1. 绝对阈值无行业上下文

**典型表现**：`ROE > 15%`、`资产负债率 < 50%`、`分红率 > 30%` 作为统一标准

**问题**：银行 ROE 10-12% 即优秀，消费公司 20%+ 很常见；银行负债率 85-95% 是商业模式本身

**修复**：
- 改用"对比行业中位数"而非固定值
- 若必须保留绝对值，添加"阈值校准说明"：注明数据来源、适用范围、校准触发条件
- 参考模板：`§3.2 行业差异说明` 的表格格式

### 2. 术语定义不完整

**典型表现**：术语表有核心概念但遗漏辅助概念（如"信源等级"的 emoji 含义）

**检查方法**：grep 所有 emoji 标记（🟢🟡🔴）和领域专有名词，确认每个都有定义

**修复**：在术语表中补充遗漏项，格式统一为 `| 术语 | 定义 | 来源 |`

### 3. 模板与流程不一致

**典型表现**：SKILL.md 中阈值已更新（如 `≤ 5`），但模板仍用旧值（`= 0`）

**检查方法**：逐项比对模板中的每个 checkbox/table 字段与 SKILL.md 对应章节

**修复**：同步修改模板，并为每个 checklist 项标注对应 Step 编号

### 4. 费雪/定性评估缺量化锚点

**典型表现**：`产品是否有足够市场潜力？`、`利润率是否足够高？` 无量化标准

**修复**：新增"量化参考锚点"列，给出行业中位数或经验阈值：
- 市场规模 > 100亿
- 研发/营收：高科技 > 10%、消费 > 3%、制造 > 2%
- 核心技术人员离职率 < 10%

### 5. 自检表无 Step 映射

**典型表现**：自检项只列"复验项"名称，不标注数据来自哪个 Step

**修复**：新增"产出 Step"列，让执行者知道去哪里找原始数据

### 6. 安全边际/估值阈值无校准说明

**典型表现**：`安全边际 > 20%` 直接引用巴菲特但未说明 A 股适用性

**修复**：添加校准说明：
- 数据来源（巴菲特/格雷厄姆原文）
- A 股调整建议（散户占比高、情绪波动大，实际可获得安全边际可能更大）
- 前置条件（护城河宽度、管理层评分与安全边际要求的联动关系）
- 校准触发条件（市场估值中枢变化时需重新校准）

### 7. 定性描述伪装成量化标准

**典型表现**：评分表"优秀标准"列写 `持续增长`、`稳定提升`、`显著改善` 等词，看起来像标准但无法客观判定

**检查方法**：grep 评分表中的"优秀标准"/"判定标准"列，找不含数字的描述性短语

**修复**：将描述改为可验证条件：
- `持续增长` → `连续3年增长，或5年CAGR>行业中位数`
- `显著降低评分` → 给出叠加扣分公式（如 `每多一个"是"额外扣2分`）
- `稳定或提升` → `近3年毛利率波动幅度<3个百分点，或趋势斜率>0`

### 8. FCF/经营现金流阈值缺行业校准

**典型表现**：`FCF/净利润 > 80%` 作为统一标准

**问题**：资本密集型行业（半导体、新能源）因扩张性资本开支大，FCF/净利润可能长期低于 50%；轻资产行业（软件、消费品）通常 >90%

**修复**：添加校准说明，注明行业分层和校准触发条件

### 9. 重叠框架未收敛

**典型表现**：同一判断维度（如护城河、管理层、估值）有 5+ 个独立框架，agent 不知道先用哪个、冲突时怎么取舍。

**检查方法**：统计 SKILL.md 中每个评估维度被多少个框架覆盖。>3个框架 = 需要收敛。

**修复**：
- 保留 1-2 个核心框架（一个快速过滤 + 一个深度评估）
- 其余框架移入 references/ 作为补充材料
- 在 SKILL.md 中用一行指引："详见 `references/xxx.md` 第Y节"

### 10. Step 重编号后引用未同步

**典型表现**：合并/重排 Step 后，正文、术语表、一票否决项、报告模板中的旧 Step 编号未更新。

**检查方法**：grep 旧 Step 编号模式（如 `Step 0\\.0`、`Step 0\\.5`、`Step 2\\.3`）

**修复**：逐个替换所有旧引用。注意术语表中的"详见 Step X.Y"最容易遗漏。

### 10b. 跨文件 Step 重编号级联失败

**典型表现**：SKILL.md 重编号后，rubric 文件（`scoring-rubric-*.md`）、calibration 文件（`special-entities-calibration.md`）、checkpoint-schema 文件全部保留旧编号。rubric 文件的 section headers 用 `## 2.1 市场空间` 格式，不匹配 `Step 4` 的 grep 模式，导致前三轮审计均遗漏。

**检测方法**：
```bash
# 找 rubric 文件中的 section headers（## N.M 格式 — 不匹配 "Step N" grep）
grep -rn '^## [0-9]\.[0-9]' references/scoring-rubric-*.md
# 找 checkpoint-schema 中的 step 文件名
grep -rn 'step[0-9]_.*\.json' references/checkpoint-schema.md
# 找 calibration 中的 Step 引用
grep -rn 'Step [0-9]\.' references/special-entities-calibration.md
```

**修复**：使用临时占位符避免级联冲突：
```bash
# Step 1: 旧编号→临时占位符（避免 4→5 然后 5→6 的级联）
sed -i 's/Step 4\.2/TEMP_52/g; s/Step 5\.2/TEMP_62/g' file.md
# Step 2: 临时占位符→新编号
sed -i 's/TEMP_52/Step 5.2/g; s/TEMP_62/Step 6.2/g' file.md
```

**案例**：jw-company-analysis v4.1.0→v4.1.3，SKILL.md 重编号后需同步 8 个文件（6 rubric + 1 calibration + 1 checkpoint-schema）。前三轮审计只修了 SKILL.md 内部，第 4 轮才发现跨文件漂移。

**关键陷阱**：
- `sed 's/Step 4/Step 5/g'` 会级联失败（先改 4→5，再改 5→6 时把刚改的也改了）
- rubric 文件的 `## 2.1` section headers 不匹配 `Step 4` grep，必须用 `grep -P '^\#+ [0-9]'` 单独检测
- checkpoint-schema 中的 JSON 文件名（如 `step5_industry.json`）也需要同步更新

### 11. 死引用文件

**典型表现**：references/ 中有文件但 SKILL.md 从未引用，或 SKILL.md 引用了已删除的文件。

**检查方法**：
1. 列出 references/ 中所有文件
2. grep SKILL.md 中的 `references/` 引用
3. 交叉比对，找出只在一侧存在的文件

**修复**：删除未引用的文件（内容已合并到 SKILL.md 或其他 references/ 文件时），或添加引用。

#### 11b. 假阳性死引用（"背景资料"模式）

**典型表现**：`comm -23` 检测显示 10+ 个"孤立"文件，但这些文件实际通过 SKILL.md 末尾的"参考资料索引表"被引用——只是不在执行步骤中被 `📚 详见 references/xxx.md` 内联引用。

**检测方法**：
```bash
# 找出 SKILL.md 内联引用的文件
inline=$(grep -oP 'references/[a-z][-a-z0-9]*\.md' SKILL.md | sed 's|references/||' | sort -u)
# 找出参考资料索引表引用的文件（通常在 SKILL.md 末尾）
indexed=$(grep -oP '[a-z][-a-z0-9]*\.md' SKILL.md | sort -u)
# 真正的死引用 = 不在索引表中也不在内联引用中
comm -23 <(ls references/*.md | xargs -I{} basename {} | sort) <(echo "$indexed")
```

**区分处理**：
- 在索引表中但不在内联中 = **背景资料**（保留，agent 可按需加载）
- 不在索引表中也不在内联中 = **真死引用**（删除或移入 `_archived/`）
- 在内联中但不在索引表中 = 不可能（索引表应覆盖所有内联引用）

**案例**：jw-company-analysis v4.1.5 审计时，`comm -23` 显示 10 个"孤立"文件，但 8 个是背景资料（investment-theory.md 9833行、zhang-xinmin-core.md 2097行等），仅 1 个是真死引用（compression-techniques-execution-content.md，移入 `_archived/`）。

## 审计分级（P0-P5）

详见 `references/large-skill-refactoring.md` 的"审计分级"章节。

**快速分级规则**：
- 引用旧Step编号/权重矛盾 → P0（立即修）
- SKILL.md > 1000行/重叠框架/术语不完整 → P1（结构性重构）
- 模板顺序不一致/校准说明分散 → P2（流程优化）
- 死引用/版本记录冗长 → P3（清理）

## 领域知识补充模式（从书籍提炼到 Skill）

> 来源：jw-company-analysis v2.3→v2.4 张新民财报分析补充经验

### 问题

投资类 Skill 的分析框架往往基于有限的知识源（如巴菲特语录），缺少本土化、行业特异性的分析维度。用户拥有完整的投资书籍库（如 /root/books/），需要将书中知识系统性地提炼补充到 Skill。

### 流程

1. **先读 SKILL.md 全文**，理解当前框架的覆盖范围和薄弱环节
2. **读两本书**（或指定书籍），提取 4 个维度的增量知识：
   - **数据维度**：新的计算公式、比率、指标（如核心利润率、资产金融负债率）
   - **调用维度**：新的分析框架（如八看框架、资本引入战略分类）
   - **分析维度**：新的判断逻辑（如经营驱动型 vs 投资驱动型分类）
   - **输出维度**：新的呈现方式（如控制性投资扩张效应的展示）
3. **三文件同步补充**：
   - SKILL.md：在对应 Step 中添加执行规则（公式、判定逻辑、阈值）
   - templates/：在对应章节中添加模板字段
   - references/：在深度指南中添加理论详解（原文引用 + 计算示例）
4. **一致性验证**：grep 新增关键词，确认在三文件中均有出现

### 并行委托修改模式

当需要同时修改 3+ 个文件时，用 `delegate_task` 并行执行：

```
delegate_task(tasks=[
  {"goal": "修改 SKILL.md Step X", "context": "具体修改内容", "toolsets": ["file"]},
  {"goal": "更新 templates/ 模板", "context": "具体修改内容", "toolsets": ["file"]},
  {"goal": "更新 references/ 深度指南", "context": "具体修改内容", "toolsets": ["file"]}
])
```

**关键**：每个子任务的 context 必须包含足够信息（当前文件内容片段、修改目标、格式要求），因为子 agent 无父对话上下文。

### 质量标准

- 新增公式必须有"计算示例"（在 references/ 中）
- 新增阈值必须有"行业差异说明"或"校准说明"
- 新增框架必须有"与现有框架的关系"说明（替代/补充/增强）
- 版本号更新 + 版本记录一条

## 审计分级补充（v3.1→v3.3 审计经验）

> 来源：jw-company-analysis v3.1.0→v3.3.0 三轮审计修复（2026-06-05）

### 12. 评分机制内部矛盾

**典型表现**：同一 Step 内有两个子节定义不同的评分机制（如 Step 5.1 定义叠加扣分 -5/-12/-21，Step 5.2 说"每'是'-5分"线性扣分）

**检查方法**：grep 同一 Step 内所有"评分标准""扣分""得分"关键词，确认机制一致

**修复**：以更精确的机制为准，另一处改为引用（如"按 5.1 叠加扣分机制"）

### 13. 流程图描述与实际内容不同步

**典型表现**：执行总览流程图写"5项数据复验"，但实际 Step 8 已扩展为 8 项

**检查方法**：比对流程图中的每个数字与对应 Step 的实际内容

**修复**：更新流程图中的数字

### 14. 内容放置在错误的 Step 之间

**典型表现**：审计报告四类意见放在 Step 0 和 Step 1 之间，但它属于财务分析范畴

**检查方法**：审查每个 Step 之间的"间隙"内容，确认其逻辑归属

**修复**：移至逻辑上属于的 Step 内部

### 15. 术语定义遗漏新增框架的核心概念

**典型表现**：新增了"核心利润""资产金融负债率"等概念，但术语表未同步添加

**检查方法**：grep 所有"（张新民）""（巴菲特）"标注的概念，确认术语表有对应条目

**修复**：补充术语定义

### 16. 速览模式子步骤缺少跳过原因说明

**典型表现**：速览模式跳过某些检查项，但未说明为何可以跳过

**检查方法**：审查速览模式每个"跳过"项，确认有简要原因

**修复**：添加"注：跳过X项，因需要Y数据/分析，速览模式不执行Z"

### 17. 复杂公式缺少快速参考表

**典型表现**：Step 2 有 30+ 个公式和判断标准，agent 执行时需频繁跳转 references

**检查方法**：统计 Step 中的公式/判断标准数量。>10个 = 需要快速参考表

**修复**：在 Step 开头添加"核心公式快速参考"表（概念 | 公式 | 优秀标准）

### 18. 大型内联 JSON/Schema 块

**典型表现**：检查点 JSON 结构占 30+ 行内联在 SKILL.md 中

**检查方法**：grep 内联代码块（```json...```），统计行数。>20行 = 应移至 references

**修复**：移至 references/xxx-schema.md，SKILL.md 保留一行引用

### 19. "常见错误" section 与 Step 内容重复

**典型表现**："常见错误与反模式"中的"决策层面"与 Step 5.1 误判心理学检查高度重复

**检查方法**：比对"常见错误"中的每个条目与对应 Step 的内容

**修复**：删除重复 section，核心内容合并到"能力边界声明"或保留一行引用

## 多轮审计收敛模式（投资类 Skill）

> 来源：jw-company-analysis v3.1→v3.27.0 五轮审计实战经验

| 轮次 | 典型问题数 | 问题类型 | 修复时间 |
|------|-----------|---------|---------|
| 第1轮 | 15-20 | 结构性问题（模板不对齐/公式错误/章节顺序/瘦身） | 60-90分钟 |
| 第2轮 | 5-10 | 一致性问题（评分机制矛盾/流程图不同步/术语遗漏） | 20-30分钟 |
| 第3轮 | 3-5 | Agent执行可行性（评分数学/引用分类/排除格式/预执行分数） | 15-20分钟 |
| 第4轮 | 1-3 | 深层漂移（结论矩阵版本漂移/操作术语未定义/ESG逻辑矛盾） | 10-15分钟 |
| 第5轮 | 8-12 | 结构性重构+边缘优化（Step膨胀瘦身/财务内容归属/阈值校准/模板行数漂移/Agent读取指引） | 30-60分钟 |
| 第6轮 | 17 | 全量审计+跨文件同步（模板终检清单清单⑤⑥⑧漂移/rubric版本号/旧Step编号残留/孤立reference评估/安全阀歧义/引用简写清理/DCF校准/术语定义/触发词扩展） | 40-50分钟 |
| 第7轮 | 4 | 微调收敛（结论矩阵emoji间距/SKILL.md压缩至≤500行/阈值标定说明/终检清单名称统一） | 10-15分钟 |

**停止信号**：连续两轮产出 0 个 P0/P1 问题时停止。

**审计工具升级**：第3轮起应使用 skill-creator 的 13 类审计清单 + 7 类优化方法论，而非仅靠 grep 结构化扫描。

**批量修复策略**：当问题数 >5 时，按文件分组用 `delegate_task` 并行执行：
- 任务1：SKILL.md 的所有 P0+P1 修改
- 任务2：模板修改 + references 新增/更新
- 任务3：版本号 + 版本记录更新

## 审计工作流（投资类 Skill 专用）

1. **版本漂移检查（第一步）**：比对 SKILL.md frontmatter 版本号与模板末尾版本号。不同步 → 模板可能是旧逻辑
2. **旧引用残留扫描**：grep 旧 Step 编号模式（`Step 0\\.0`、`Step 0\\.5`、`Step [0-9]\\\\.[0-9]`）确认无残留
3. **权重表唯一性**：grep `权重` 确认只出现一次完整表格
4. **术语完整性**：grep 所有 emoji 和领域术语 → 逐个检查术语表
5. **阈值一致性**：grep 所有数字阈值 → 检查是否有行业差异说明
6. **重叠框架计数**：统计每个评估维度被几个框架覆盖 → >3个需收敛
7. **模板同步（深度）**：不仅比对数值，还要比对：
   - 表格列数和列名（SKILL.md 的4列模板是否在模板中被简化为3列？）
   - 公式是否存在（SKILL.md 有计算公式但模板只有占位符？）
   - checklist 项的 Step 编号引用是否正确
   - 报告模板中的 Step 编号 = 执行顺序
8. **自检映射**：检查自检表是否有"产出 Step"列
9. **校准说明**：检查绝对阈值是否有校准元数据
10. **定性描述扫描**：grep 评分表"优秀标准"列，找无数字的描述性短语（见问题 #7）
11. **死引用检查**：列出 references/ 文件 × SKILL.md 引用，交叉比对
12. **三文件一致性验证**：每次修改后，用 grep 检查关键词在 SKILL.md、templates/、references/ 三处是否同步出现。不一致 = 遗漏。

## 问题 #20-#23：Agent执行可行性（Round 4+审计）

> 来源：jw-company-analysis v3.17.1→v3.18.0 第三轮审计（2026-06-03）
> 这类问题在前三轮审计中不会出现，只有当用户问"agent能执行吗"时才会暴露。

### 20. 评分体系数学不一致

**典型表现**：结论矩阵定义80-100=优秀，但实际最大得分=90(加权)+5(安全阀)=95。agent查表时发现大多数公司落在50-69区间。

**检查方法**：
```bash
# 计算实际最大/最小得分
echo "最大=100×35%+100×20%+100×10%+100×25%+5" | bc
# 对比结论矩阵区间
grep -A1 '80-100\|70-79' SKILL.md
```

**修复**：根据实际得分分布调整矩阵区间，添加分数校准说明。

### 21. "详见"引用agent不会读

**典型表现**：SKILL.md有50+处"详见 references/xxx.md §YY"，reference文件有2000+行。agent在8-15分钟执行时间内不可能读完。

**检查方法**：
```bash
grep -c '详见.*references/' SKILL.md
wc -l references/*.md | sort -rn | head -5
```

**修复**：引用分类——SKILL.md内联(agent必读) / 📚人类参考 / 🔧脚本工具。

### 22. 排除检查点格式不统一

**典型表现**：流程图中排除检查有的有量化阈值（"经营现金流/净利润<1.0"），有的是主观判断（"四问检验不清"），有的后果不明确（"不通过=降级"还是"排除"？）。

**修复**：统一格式为 `排除检查：[名称]（[客观条件]+[量化阈值]）→ ❌排除`。

### 23. 预执行脚本默认分数误导

**典型表现**：pre_analysis.py给管理层评分默认15分、护城河默认12分。agent直接采用而不做定性分析。

**修复**：给每个默认分数添加`_note`字段标注"⚠️待agent确认: [需要做的定性分析]"。

## 问题 #24-#29：深度审计补充（Round 5+ 审计）

> 来源：jw-company-analysis v3.27.0 第五轮审计（2026-06-04）
> 使用 skill-creator 的 13 类审计清单 + 7 类优化方法论系统检查发现。

### 24. 结论矩阵在 SKILL.md 与模板之间版本漂移

**典型表现**：SKILL.md 定义 4 档结论矩阵（≥80/65-79/50-64/<50），模板定义 6 档（85-92/75-84/65-74/55-64/40-54/<40）。Agent 填模板和算结论用不同标准。

**检测方法**：
```bash
# 比对 SKILL.md 和模板的结论矩阵行数
grep -c "优秀\|好标的\|观望\|不推荐\|远离" SKILL.md
grep -c "优秀\|好标的\|一般\|观望\|不推荐\|远离" templates/*.md
```

**修复**：以 SKILL.md 为准，同步模板。矩阵行数必须一致。

#### 24b. 结论矩阵 Emoji 间距/措辞微漂移

**典型表现**：矩阵行数一致但 emoji 后缺空格或措辞微差。如 SKILL.md 写 `⭐ 优秀`，模板写 `⭐优秀标的`（emoji 后无空格 + 多了"标的"二字）。肉眼难以发现，但 agent 填模板时用 SKILL.md 的措辞，导致报告前后不一致。

**检测方法**：
```bash
# 逐行比对结论矩阵的每个单元格
diff <(grep -A4 '投资结论矩阵' SKILL.md | grep '|') \
     <(grep -A4 '投资结论矩阵' templates/*.md | grep '|')
```

**修复**：以 SKILL.md 为准，逐字符同步模板（包括 emoji 后的空格）。

**案例**：jw-company-analysis v4.1.5 审计时发现模板 `⭐优秀标的` vs SKILL.md `⭐ 优秀`，差异仅 1 个空格 + 2 个汉字，但会影响报告一致性。

### 25. 操作性术语未定义（"降级"）

**典型表现**：排除检查中多次使用"⚠️降级"但从未定义：降级=扣多少分？降结论等级？进入特别关注清单？

**检测方法**：
```bash
grep -n "降级" SKILL.md
# 如果出现 3+ 次但无定义块 → 命中
```

**修复**：在首次出现"降级"前增加定义块：
```markdown
**"降级"定义**：综合结论下调一档（如"优秀"→"好标的"）+ 报告中列为⚠️关注项。
降级不同于排除（❌）：降级后仍可输出结论，但结论等级受限。
```

### 26. 流程图数字精度问题

**典型表现**：执行总览流程图写"安全阀±5分"，但实际 Step 5 原始范围 -50到+20。"±5分"是加权后的值，流程图给人错误印象。

**检测方法**：比对流程图中每个数字与对应 Step 的实际定义范围。

**修复**：流程图中的数字应与 Step 定义一致，加权值用括号注明。

### 27. ESG 逻辑路径矛盾

**典型表现**：流程图写"ESG→❌排除"（不可恢复的硬排除），但 ESG section 描述扣分机制（0/-5/-10）。排除和扣分是两个不同逻辑路径，同一次检查不可能同时走两条路。

**检测方法**：
```bash
grep -n "ESG" SKILL.md
# 检查是否有"排除"和"扣分"两种不同后果描述
```

**修复**：明确分离两条路径：
- ESG 排除条件（硬排除，触发任一→❌）：审计非标/质押>80%/实控人被查/重大安全事故
- ESG 风险评估（扣分，不触发排除时）：低+0/中-5/高-10

### 28. Step 0.5 "快速扫描"膨胀为巨型参考手册

**典型表现**：Step 0.5 设计为"分析前快速扫描宏观环境"，但实际膨胀到 167 行、6 个子模块，内容比 Step 1-4 还多。全部是"详见 references/"引用，agent 在 8-15 分钟执行时间内不可能读完。

**检测方法**：
```bash
# 统计每个 Step 的行数
awk '/^## Step|^### 0\./{if(s)print s, NR-b; s=$0; b=NR} END{print s, NR-b}' SKILL.md
# 如果 Step 0.5 行数 > Step 1 行数 → 膨胀
```

**修复**：
- SKILL.md 中 Step 0.5 精简为"快速扫描清单"（3-5 个关键判断点）
- 详细框架移入 `references/macro-context.md`
- SKILL.md 保留一行指引

### 29. 引用分类指引缺失

**典型表现**：Skill 有引用分类系统（SKILL.md 内联/📚人类参考/🔧脚本工具），但没有在概览中明确告诉 agent"执行时只需读 SKILL.md 内联内容"。151 处"详见"引用可能让 agent 误以为都需要读。

**检测方法**：
```bash
grep -c '详见.*references/' SKILL.md
# >100 且概览无"agent只需读内联"指引 → 命中
```

**修复**：在概览的"📖 引用分类说明"后增加：
```
**Agent 执行原则**：执行时只需读 SKILL.md 内联内容。📚人类参考仅供人类深入研究，agent 不需要在执行时间内读取。
```

### 30. 财务分析内容漂移到逆向检查 Step

**典型表现**：多轮迭代后，财务风险分析内容（如"财务爆雷七大区域"）被放在 Step 5（逆向检查与误判防护），而非 Step 2（财务健康度）。原因是 Step 5 在多轮迭代中成为"catch-all section"——每轮新增的逆向/风险相关内容都堆在 Step 5，最终包含了本应属于其他 Step 的内容。

**检测方法**：
```bash
# 统计每个 Step 中"详见"引用的数量
for step in "Step 0" "Step 1" "Step 2" "Step 3" "Step 4" "Step 5" "Step 6" "Step 7"; do
  echo "$step: $(sed -n "/$step/,/^## Step/p" SKILL.md | grep -c '详见')"
done
# 如果某个 Step 的引用数是其他 Step 的 3 倍以上 → catch-all
```

**修复**：将内容移至逻辑上归属的 Step。在原位置保留一行交叉引用："详见 Step X.Y（已移至更合理的归属）"。

### 31. 阈值校准说明缺失（第5轮深度审计）

**典型表现**：绝对阈值（如`资产金融负债率<20%优秀`、`商誉/净资产>30%红线`）没有说明数据来源、适用时间范围、或校准触发条件。与 #1（绝对阈值无行业上下文）不同，这里是"有行业校准但无数据来源校准"。

**检测方法**：
```bash
grep -n "< [0-9]*%\|> [0-9]*%\|红线" SKILL.md
# 检查每个阈值附近是否有"校准说明"或"基于"字样
```

**修复**：为每个关键阈值添加校准说明，格式：
```
（校准说明：基于[数据来源][时间范围]，[适用条件]需用[校准系数]调整。）
```

## 审计工作流补充（第5轮经验）

在已有审计工作流的第12步之后，增加第13步：

13. **Step 引用密度分析**：统计每个 Step 中的"详见"引用数量。如果某个 Step 的引用密度是其他 Step 的 3 倍以上，该 Step 很可能已成为"catch-all"——需要检查其中是否有内容应归属其他 Step。

## 问题 #32-#35：模板深层漂移（Round 8+ 审计）

> 来源：jw-company-analysis v3.28.0 第八轮审计（2026-06-03）
> 这类问题在前几轮审计中不会出现，只有当模板与SKILL.md的结构差异足够大时才暴露。

### 32. 模板排除条件列表缺失

**典型表现**：SKILL.md 定义了明确的排除条件列表（如 ESG 排除4条件：审计非标/质押>80%/实控人被查/重大安全事故），但模板只写"高风险-10分**或排除**"——未列出具体条件。Agent 执行时可能只做扣分而不触发排除。

**检测方法**：
```bash
# 找SKILL.md中的排除条件定义
grep -n '排除条件\|直接排除.*：\|排除（触发' SKILL.md
# 对比模板中是否有对应条件列表
grep -n '排除' templates/*.md
```

**修复**：在模板中补充排除条件完整列表，与SKILL.md一字不差。

### 33. 新增Step输出区域缺失

**典型表现**：SKILL.md 新增了 Step 0.5（宏观环境评估）并定义了输出格式（step0_macro.json），但模板中完全没有对应的记录区域。Agent 执行后无处填写宏观评估结果。

**检测方法**：
```bash
# 列出SKILL.md中所有Step及其输出文件
grep -n '输出.*json\|→ 输出' SKILL.md
# 对比模板中是否有对应的记录区域
grep -n 'step[0-9]_.*\.json\|宏观环境\|Step 0\.5' templates/*.md
```

**修复**：在模板中插入新Step的输出区域——表头说明数据来源 + 关键字段表格 + 状态标记。

### 34. 条件调节规则缺失

**典型表现**：SKILL.md 定义了条件调节规则（如技术面信号→建仓比例调节：🟢🟢上调至40%/🟡下调至20%/🔴暂停），但模板只有固定值（30%/50%/70%）。Agent 无法体现调节逻辑。

**检测方法**：
```bash
# 找SKILL.md中的条件调节/联动规则
grep -n '上调\|下调\|偏积极\|偏保守\|暂停' SKILL.md
# 对比模板中是否有对应调节逻辑
grep -n '上调\|下调\|偏积极\|偏保守' templates/*.md
```

**修复**：在模板的对应位置补充条件调节规则表或交叉引用。

### 35. 预执行脚本版本落后

**典型表现**：pre_analysis.py 版本号为 v3.22.0，SKILL.md 已到 v3.28.0。6个版本未同步，可能有评分逻辑/输出格式不一致。此问题在"版本号四端同步"检查中容易被遗漏——因为四端不包含脚本版本。

**检测方法**：
```bash
# 检查脚本版本
grep -n '版本\|version' scripts/*.py | head -5
# 对比SKILL.md版本
head -3 SKILL.md
```

**修复**：更新脚本版本号。如果脚本有硬编码的评分逻辑或输出格式，需逐版本检查变更并同步。投资类skill的版本同步应为五端（含预执行脚本）。

## 问题 #36-#38：执行流程序列化审计（Round 9+ 审计）

> 来源：jw-company-analysis v4.0.0 第四轮审计（2026-06-05）
> 从内容结构×执行逻辑×执行步骤三个维度发现。

### 36. 检查点文件名与Step重编号不同步

**典型表现**：Step重编号后（如Step 1→Step 3），pre_analysis.py仍生成`step1_quality.json`，但SKILL.md期望`step3_business.json`。checkpoint恢复时静默失败——agent找不到文件，从Step 0重新执行，浪费预执行节省的时间。

**检测方法**：
```bash
# 列出脚本中所有step文件名引用
grep -n 'step[0-9]_' scripts/*.py
# 列出SKILL.md中期望的文件名
grep -n 'step[0-9]_.*\.json' SKILL.md
# 交叉比对，找不匹配的
```

**修复**：
1. 更新脚本中的`STEP_FILE_MAP`字典
2. 更新脚本注释中的文件名列表
3. 更新SKILL.md中"预执行与检查点"的文件名描述
4. 更新checkpoint-schema.md中的文件名和JSON结构
5. **五端验证**：脚本、SKILL.md、checkpoint-schema.md、模板、version-history.md中的step文件名必须一致

### 37. 复合评分公式范围溢出

**典型表现**：安全阀定义"范围-50到+20"，但实际计算中四维风险(0到-40)+逆向(±10)+情绪(±10)+ESG(0/-5/-10)=-60到+20。文档中的范围比实际窄10分。agent按文档判断"得分<-20触发复核"时，可能漏掉-60到-50区间的情况。

**检测方法**：
```bash
# 找到评分公式
grep -A5 '总分.*=.*扣分.*+.*检查' SKILL.md
# 手动计算各分项的最小值之和
# 对比文档中声明的范围
grep '范围.*-50\|范围.*-60' SKILL.md
```

**修复**：重新计算各分项的实际最小值和最大值，更新文档中的范围声明。如果发现溢出，要么扩大文档范围，要么调整分项权重使实际范围匹配文档。

### 38. Step间依赖关系未显式声明

**典型表现**：每个Step只写"输入：stepX_data.json"，但实际依赖更复杂（如Step 6估值需要Step 2行业增速+Step 3商业模式+Step 5财务数据）。agent执行Step 6时可能遗漏Step 2/3的分析结论，导致DCF假设缺乏行业/商业模式上下文。

**检测方法**：
```bash
# 列出每个Step的输入声明
grep -n '输入.*输出' SKILL.md
# 检查是否有"依赖"声明
grep -c '依赖.*Step' SKILL.md
# 如果依赖声明数 < Step数的一半 → 大部分Step缺少依赖声明
```

**修复**：在每个Step的输入声明后增加`> **依赖**：Step X（数据）、Step Y（分析结论）`。依赖声明应列出所有直接依赖的Step，不仅限于数据输入。
