# Agent执行可行性审计模式

> 来源：jw-company-analysis v3.17.1→v3.18.0 第三轮审计（2026-06-03）
> 当skill有复杂的多步骤流程时，必须检查agent实际执行时是否会遇到障碍。

## 核心问题

SKILL.md写给人看和写给agent执行是两回事。以下问题在"人读OK但agent执行会卡"的情况下最常出现。

## 问题1：大量"详见"引用指向巨型文件

**典型表现**：SKILL.md有73处"详见 references/investment-theory.md §XX"，而该文件有5545行。agent不可能在8-15分钟内读完这些引用。

**检查方法**：
```bash
grep -c '详见.*references/' SKILL.md
wc -l references/*.md | sort -rn | head -5
```

**修复**：
- **核心判断标准**（评分公式、排除条件、阈值）必须内联到SKILL.md
- **详细理论背景**标注"📚人类参考"，说明agent不需要读
- **脚本工具**标注"🔧脚本工具"，说明agent直接调用

**引用分类模板**（在SKILL.md概览区添加）：
```markdown
**📖 引用分类说明**：
- **SKILL.md内联**：agent执行必需的判断标准、评分公式、排除条件。agent直接使用。
- **📚人类参考**：标注`详见 references/...`的引用，指向详细理论背景。agent不需要读，人类深入研究时查阅。
- **🔧脚本工具**：pre_analysis.py等自动化脚本。agent调用执行。
```

## 问题2：排除检查点格式不统一

**典型表现**：
- "0.3重大负面事件？→ ❌排除"（无量化标准）
- "1.1四问检验不清？→ ❌排除"（主观判断）
- "2.1利润三大前提不通过？→ ❌排除"（不通过=降级还是排除？）

**检查方法**：grep执行总览流程图中的所有"排除"检查，逐个验证是否有客观条件+量化阈值+后果。

**修复**：统一格式为：
```
→ 排除检查：[检查名称]（[客观条件]+[量化阈值]）→ ❌排除
```
示例：
```
→ 排除检查：0.3重大负面事件（🔴重大影响>10%营收或一票否决）→ ❌排除
→ 排除检查：1.1四问检验（Q1-Q4任一答不清）→ ❌排除
→ 排除检查：2.1利润三大前提之①利润为真（经营现金流/净利润<1.0）→ ❌排除
```

## 问题3：评分体系数学不一致

**典型表现**：投资结论矩阵定义80-100=优秀标的，但实际最大得分=90(四维加权)+5(逆向)=95。agent在Step 7查表时发现大多数公司落在50-69区间，80+几乎不可能。

**检查方法**：
1. 计算实际最大/最小得分 = Σ(维度满分×权重) ± 安全阀调整
2. 对比结论矩阵的分数区间
3. 验证典型公司得分分布是否合理

**修复**：
- 根据实际得分分布调整矩阵区间
- 添加分数校准说明：`> 综合得分 = 四维加权(满分90) + 逆向调整(±5)。实际范围约25-95分。`

## 问题4：pre_analysis.py默认分数误导agent

**典型表现**：预执行脚本给管理层评分默认15分、护城河默认12分。agent可能直接采用而不做定性分析。

**检查方法**：读取pre_analysis.py，找硬编码的默认分数。

**修复**：给每个默认分数添加`_note`字段：
```python
result["scores"]["1.3_管理层"] = 15
result["scores"]["1.3_管理层_note"] = "⚠️待agent确认: 需web搜索管理层言论+荒岛两问+三重测试"
```

## 问题5：模板数据表需要手动填写

**典型表现**：模板有12行关键数据表（股价、市值、PE、ROE等），agent需要从step0_data.json手动复制每个值。

**检查方法**：统计模板中的`[X]`占位符数量。

**修复**：在模板数据表头部添加自动填充说明：
```markdown
> **自动填充说明**：以下表格数据应从step0_data.json中的sources.quote和sources.financial字段自动提取。
```

## 问题6：环境验证只检查文件存在

**典型表现**：Step 0检查脚本文件是否存在，但不验证是否可执行、依赖是否安装。

**检查方法**：读取Step 0环境验证段，确认是否有执行验证。

**修复**：增加验证步骤：
```markdown
- 验证脚本可执行：`python <script> --help` 成功返回（exit code 0）
- 验证依赖安装：`pip list | grep -E 'yfinance|akshare|pandas'` 至少包含核心依赖
```

## 问题7：评分范围不对称（Score Range Asymmetry）

**典型表现**：Step 5标注"±50分"，但实际正向最大只能+20（误判扣分0 + 逆向检查+10 + 市场情绪+10 + ESG 0），负向最大-50（误判-50 + 逆向-10 + 市场情绪-10 + ESG-10 → 截断到-50）。文档说"±X"但实际是不对称的。

**检查方法**：
```python
# 对每个安全阀/调整维度，计算实际正向最大和负向最大
positive_max = sum(每项正向最大)
negative_max = sum(每项负向最大)  # 注意截断
print(f"实际范围: {negative_max} 到 {positive_max}")
# 对比文档声称的 "±X"
```

**修复**：
- 将"±50"改为"实际范围-50到+20"
- 更新加权影响：×10% = -5到+2（非±5）
- 更新结论矩阵最大值：基础90 + 正向调整2 = 92（非95）
- 在Step 5组合公式中明确每项的正向/负向范围

**jw-company-analysis实测**：
- 文档: "±50分制，加权影响±5分"
- 实际: -50到+20，加权-5到+2
- 结论矩阵: 85-95 → 修正为85-92
- 影响: 4处引用需修正

## 问题8：预执行脚本 scores/notes 混合导致 sum 崩溃

**典型表现**：pre_analysis.py 将评分和备注存在同一个 dict 中（如 `result["scores"]["1.1_商业模式"] = 18` 和 `result["scores"]["1.1_商业模式_note"] = "⚠️待确认"`），然后用 `sum(result["scores"].values())` 求和。Python 的 `sum()` 对字符串求和抛 `TypeError`。

**检查方法**：
```bash
grep -n 'scores.*_note\|scores.*note' <script>.py
grep -n 'sum.*scores' <script>.py
```

**修复**：将 scores dict 拆分为两个独立 dict：
```python
result = {"scores": {}, "notes": {}}  # 数值和字符串分开
result["scores"]["1.1_商业模式"] = 18
result["notes"]["1.1_商业模式"] = "⚠️待agent确认: 四问检验法"
total = sum(result["scores"].values())  # 安全
```

**关键教训**：此 bug 意味着预执行脚本从未成功运行过。但 `--help` 验证和语法检查都通过。**验证预执行脚本必须实际运行一次**，不能只检查文件存在。

## 问题9：结论矩阵"僵尸区间"

**典型表现**：结论矩阵定义 6 档分数区间，但最低 2 档在实际中永远不可达——因为公司必须先通过 Step 0 排除检查、Step 1 四问检验、Step 2 利润三大前提。这些前置筛选确保了最低四维得分不会低于 ~50 分。

**检查方法**：
1. 列出所有前置排除/降级条件
2. 估算通过所有前置条件后的最低可能四维得分
3. 对比结论矩阵的最低分数区间
4. 如果最低分数区间 < 估算最低得分 → 僵尸区间

**修复**：合并僵尸区间。例如 6 档→4 档（<50 / 50-64 / 65-79 / ≥80）。

## 问题10：评分修正后残留引用未同步

**典型表现**：修正了评分范围（如 ±50 → -50到+20）后，SKILL.md 中仍有 3-5 处旧值残留（"满分 ±50 分"、"加权仅 ±5 分"、"85-95分区间"等变体表述）。

**检查方法**：修正后全文 grep 旧值的所有变体：
```bash
grep -n '±50\|85-95\|±5 分' SKILL.md
```

**修复**：每修正一个数值/范围，必须 grep 所有变体表述并全部更新。常见变体：数值本身、中文描述、公式注释、结论矩阵引用、流程图标注。

## 审计清单（Agent执行可行性）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | "详见"引用数 vs reference文件行数 | >50处引用+>2000行文件 = 需分类 |
| 2 | 排除检查点格式统一 | 每个排除检查有客观条件+量化阈值+后果 |
| 3 | 评分体系数学验证 | 最大/最小得分与结论矩阵区间匹配 |
| 4 | 预执行脚本默认分数 | 每个默认分数有"待确认"标记 |
| 5 | 模板数据表填充方式 | 有自动填充说明或明确的手动填写指引 |
| 6 | 环境验证完整性 | 文件存在+可执行+依赖安装 |
| 7 | 评分范围对称性 | 文档声称的±X与实际正向/负向最大值一致 |
| 8 | 预执行脚本 scores/notes 分离 | scores dict 仅含数值，sum() 不会崩溃 |
| 9 | 结论矩阵僵尸区间 | 最低分数档位 ≥ 前置筛选后的最低可能得分 |
| 10 | 评分修正残留引用 | 修正数值后 grep 所有变体表述 |
