# content-summary-jw — 书籍内容蒸馏总结

## TL;DR 执行速查

| 阶段 | 核心动作 | 输入 | 输出 | 闸门 |
|---|---|---|---|---|
| **1** | 整书理解 (Adler 四步 + 类型判定) | 书本文本 | `stage1-understanding.md` | ⏸️ 用户确认 |
| **2** | 5 extractor 并行提取 | 文本 + 阶段1理解 | `candidates/*.md` | — |
| **3** | 三重验证 + 优先级评分 | `candidates/` | `verified.md` + `rejected/` | ⏸️ 用户确认 + 异常闸门 |
| **4** | 撰写 SUMMARY + INDEX | `verified.md` | `SUMMARY.md` + `INDEX.md` | ✅ 自检后交付 |

> 详细执行方法论见 `methodology/` 目录。遇到边界情况先查 SKILL.md 的异常处理表，再查 methodology 文件。

## 使命

把一本书里的核心方法论、框架、原则、案例、反例和术语，提炼成结构化的总结报告，让人读完后能真正理解并在实际场景中运用。

**边界**:
- ✅ 做: 方法论提炼 / 框架总结 / 原则提取 / 概念体系整理 / 书摘
- ❌ 不做: 作者人设角色扮演
- ⚠️ 适用范围: 非虚构方法论类书籍（商业/心理/哲学/自我提升/学术）。**纯虚构作品（小说/戏剧/诗歌）不适用**——这些书的"思想"是文学性的，无法用 R/I/A1/A2/E/B 六段蒸馏。如用户坚持处理虚构作品，只输出 Adler 四问导读（INDEX.md），不生成 SUMMARY.md 方法论单元

## 核心方法论: 四阶段流水线

```
阶段 1: 整书理解 (Adler 分析阅读) + 阶段产出持久化
阶段 2: 并行提取 → 候选方法论单元池 (candidates/)
阶段 3: 三重验证筛选 → 优先级评分 → verified.md
阶段 4: 结构化总结撰写 → SUMMARY.md + INDEX.md
```

## 何时调用此 skill

用户说类似:
- "帮我总结《穷查理宝典》"
- "拆解《矛盾论》的方法论"
- "提炼这本书的核心框架"
- "这本书在讲什么"
- "summarize this book: <path>"

## 输入要求

在开始前**必须**从用户处确认:
1. **书的文本来源**: PDF / EPUB / TXT 文件路径，或可访问的纯文本。**不要**在没有文本的情况下"凭记忆"总结。
2. **书名 + 作者 + 出版年**: 用于目录命名和审计。

### 文本来源异常处理

| 异常 | 处理 |
|---|---|
| 文件不可读（路径不存在/权限不足） | 告知用户具体错误，请用户重新提供路径 |
| DRM 保护 / 加密文件 | 告知"该文件受 DRM 保护，无法提取文本"，请用户提供未加密版本或手工粘贴关键章节 |
| 扫描件 PDF（无文字层） | 告知可尝试 OCR，但准确率不保证；建议用户提供纯文本版 |
| 文件格式不支持（如 .mobi / .azw3） | 建议用 calibre 等工具转换为 EPUB 或 TXT 后重试 |
| 单阶段执行失败 | 将该阶段的已有成果持久化写入文件；告知用户失败位置和原因；**由用户决定继续重试还是跳过** |
| 阶段 3 异常（通过率 <5% / >80% / 单元 >30） | 触发**闸门 3**（见下方闸门表），暂停等高知用户决策 |

## 重要区分：阶段 1 的四问 vs 阶段 4 的四问

两者形式相同，目的不同，**不得混淆**:

| | 阶段 1 的四问（Adler 分析阅读） | 阶段 4 的四问（输出结构） |
|---|---|---|
| **目的** | 理解策略：读者向自己提问 | 导读策略：向读者展示概览 |
| **"有道理吗"** | 批判性评价：写入 `stage1-understanding.md` 的批判段 | 概述性评价：写在 SUMMARY 开头 |
| **位置** | agent 上下文内，不出现在输出中 | 出现在 SUMMARY.md 和 INDEX.md 前置位置 |

**操作要点**: 阶段 1 的批判性评价不得混入阶段 4 的四问输出。

## 输出结构

```
books/<book-slug>/
├── stage1-understanding.md    # 阶段 1 产出: 整书理解的持久化记录
├── INDEX.md                   # 阶段 4 产出: 全书导航 + 元信息 + 四问摘要
├── SUMMARY.md                 # 阶段 4 产出: 核心方法论 + 四问展开 + 审计
├── candidates/                # 阶段 2 产出: 原始候选池 (审计用)
│   ├── frameworks.md
│   ├── principles.md
│   ├── cases.md
│   ├── counter-examples.md
│   └── glossary.md
├── verified.md                # 阶段 3 产出: 通过验证的单元 + 优先级评分
└── rejected/                  # 阶段 3 产出: 未通过验证的单元 (含原因)
    └── <id>.md
```

## 书籍分类与 extractor 策略

**阶段 1 末尾必须完成书籍类型判断**，据此决定各 extractor 的处理权重:

| 书籍类型 | 特征 | 重点 extractor | 可简化的 extractor |
|---|---|---|---|
| 方法论专著 | 大量框架/原则/推理 | 全部认真跑 | — |
| 散文/随笔集 | 碎片化洞见，少体系 | framework + principle | case（轻量） |
| 学术/理论著作 | 术语密集，论证严密 | glossary + framework | — |
| 传记/案例集 | 大量真实事件 | case 为主 | framework（轻量） |
| 虚构作品（小说/戏剧/诗歌） | 无方法论骨架，思想是文学性的 | **不适用**：输出 Index.md 四问导读，不生成 SUMMARY.md | — |
| 其他（画册/诗集/菜谱） | 无文本论证结构 | **不适用**：告知用户本书不适合方法论提炼 | — |

**书籍类型判定后**: 如判定为"不适用"，立即告知用户原因并给出替代方案（INDEX.md 导读或终止），**不进入阶段 2-4**。

## 长书策略

当书籍文本超过 5 万字，或预估 token 消耗可能溢出上下文窗口时:

1. **分块**: 按章节将书籍切分为独立文本块，每块独立运行 extractor
2. **合并**: 各块候选池合并去重后，再进入阶段 3
3. **阶段 1 先行**: 先完成阶段 1 的全书理解（结构/术语/主线），再按块执行阶段 2，避免在局部文本上耗费 token

## 质量红线 (违反则阻止输出)

1. 每个方法论单元必须有**原文引用**，标注出处章节
2. 每个单元必须有完整的 R / I / A1 / A2 / E / B 六段
3. 原文引用 ≤150 字/段
4. 总结必须体现作者**独特的反直觉见解**，不是常识复述

## 交卷前自检清单（阶段 4 末尾必须执行）

在输出 SUMMARY.md 之前，逐项确认:

- [ ] **红线 1**: 每个方法论单元都有原文引用，标注了章节位置
- [ ] **红线 2**: 每个方法论单元都有 R/I/A1/A2/E/B 六段，缺一不可
- [ ] **红线 3**: 所有原文引用单段 ≤150 字
- [ ] **红线 4**: 总结体现了作者独特的反直觉见解，不是普通常识
- [ ] **四问板块**: 一至四每问都已填写，无占位符残留
- [ ] **锚点一致**: SUMMARY.md 中每个方法论的标题即为目录锚点，保持完全一致
- [ ] **INDEX/SUMMARY 分工**: INDEX 不重复 SUMMARY 的详细四问展开内容，只保留摘要

## 调用惯例

- **主动汇报**: 每阶段完成后报告进度，不静默跑完 dump 结果
- **不凭记忆**: 无文本就停下来向用户要
- **保留审计**: candidates/ 和 verified.md 都要持久化
- **并行优先**: 阶段 2 的 5 extractor 并行运行（详见 `methodology/02`），不行则串行回退（互不读取彼此输出）

### 用户确认闸门（必须执行）

| 闸门 | 位置 | 操作 | 为什么 |
|---|---|---|---|
| **闸门 1** | 阶段 1 完成后 | 展示理解摘要（类型/主旨/核心命题/单元预估值）；**等确认再进阶段 2** | 理解偏差此时纠正成本最低 |
| **闸门 2** | 阶段 3 完成后 | 展示验证摘要（通过/淘汰数、Top 5 及评分）；**等确认再进阶段 4** | 三重验证取舍需用户判断 |
| **闸门 3** | 阶段 3 异常（通过率 <5%/>80%/单元 >30） | 通过率异常→extractor或验证问题；单元过多→建议 Top-15 或拆分。**不等确认不继续** | 异常不解决则输出不可用 |

**闸门通行规则**: 用户说"继续"/"ok"/"没问题"即可通过。用户在闸门处叫停时，说明具体原因，回到对应阶段修正。
