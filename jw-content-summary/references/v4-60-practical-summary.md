## 5 个最常中招的陷阱（高频模式）

### 1. v4.58 Pitfall 12：I 段后接"列表"被 Format B 误捕获（8/8 全部中招）

**触发场景**：12 单元 SUMMARY.md 写完后，validator 报 `[OVER] I-section unit 1 self-narration: 400-700 chars (limit 300)`——但实际 Unit 1 I 段只有 70-200 字符，**"超长"的不是 Unit 1 而是其他 unit**。

**根因**：Format A 抓不到（其他 11 unit 是 inline 格式，只匹配 1 个）→ fallback Format B 贪心抓取从该 unit 的 `**I — Interpretation**：` 到 `\n\n**A1` 之间的所有内容（包含列表）。

**修复策略**（按优先级）：
1. **首选**：将列表移到 A1 之前作为独立子段（用 H4 + 列表）+ 压缩为单行格式
2. **次选**：将列表内容重组到 A1 历史案例中（最稳）
3. **次选（最激进）**：删除"列表"子段——如果信息已隐含在 I 段论述中

**实战案例**：
- 《投资中最简单的事》Unit 6/7/8/9/11/12 **6 个 I 段**全部超长 400-700 字符（每个 I 段后接 6-9 类陷阱列表 + 案例列表）
- 《投资中不简单的事》Unit 10 I 段后接"7 新特点"列表 + Unit 12 I 段后接"5 投资方法"列表
- 《巴菲特之道》Unit 12 I 段后接"3 大支柱 5 维度"列表

**批量修复脚本**（v4.60 新增）：
```bash
python3 scripts/fix_overlong_i_sections.py SUMMARY.md
# 自动扫描所有 I 段 + A1 中间块，超 200 字符的截断到 200 字符
```

**预防**（v4.60 最佳实践）：**写 SUMMARY 时**直接用 v4.58 推荐的 3 个修复策略之一作为**初始写法**——即将列表内容设计在 A1 段（不超 300 字符限制），而非 I 段后。

### 2. v4.60 Pitfall：章节引用格式错误（8/8 全部中招）

**触发场景**：validator 报 `[MISSING] No R-section chapter citations found`。

**根因**：
- 章节引用**不以"第N章"开头**——`> ——邱国鹭，第1章` → 0 匹配
- 章节引用**有空格**——`> ——第 1 章` → 0 匹配
- validator 正则：`r'> ——\s*第[零一二三四五六七八九十0-9篇章节]+'`

**修复脚本**（v4.60 新增）：
```bash
python3 scripts/fix_chapter_citations.py SUMMARY.md
# 自动检测 + 修复 4 种常见错误格式
# 1. > ——<作者>/<时间>，<职位>，第N章 → > ——第N章（<作者>）
# 2. > ——<作者>，<时间>，第N章 → > ——第N章（<作者>）
# 3. > ——<作者>，第N章 → > ——第N章（<作者>）  ← 最常见
# 4. 第 N 章 → 第N章 （去空格）
```

**正确格式**：
```markdown
> ——第1章（邱国鹭）      ← 推荐
> ——第1章               ← 最简
> ——第N讲 / 第N部分 / 第N篇
```

**注意**：**章节引用在引用行末尾独立成行**（v4.26 Pitfall 3 + v4.43 强化）——不能写成同行格式 `> 通常，投资者...(引文内容)> ——（第五章）"`——必须拆成 `> 引文内容\n> ——第五章`。

### 3. v4.44 五问 phrase 严格匹配（5/8 中招）

**触发场景**：validator 报 `[MISSING] Five-question standard format, missing: ['作者的写作意图', '这本书在谈什么', ...]`

**根因**：validate 用 phrase list 做 substring 检测——H2 标题必须**包含**标准 phrase，**不能**只是"传达相同含义"。

**正确格式**（PASS）：
```markdown
## 〇、作者的写作意图
## 一、这本书在谈什么（核心主张）   ← 包含 phrase "这本书在谈什么"
## 二、全书的内容结构              ← 包含 phrase "全书的内容结构"
## 三、这本书有道理吗 / 边界在哪    ← 包含 phrase "这本书有道理吗"
## 四、这本书与自己的关系          ← 包含 phrase "这本书与自己的关系"
```

**错误格式**（FAIL）：
```markdown
## 一、核心主张是什么？            ← 不含任何标准 phrase
## 二、全书结构是怎样的？           ← "内容结构"是 "全书的内容结构" 子串，但 phrase "内容结构" 在末尾
## 三、这本书有道理吗和边界        ← "有道理吗"匹配，但空格问题
```

**5 个 phrase 词表**（v4.44 固定）：
1. `作者的写作意图`
2. `这本书在谈什么`
3. `全书的内容结构`
4. `这本书有道理吗`
5. `这本书与自己的关系`

### 5. v4.51 Pitfall 6：R-block 长度检查被 H4 标题绕过（预防性）

**触发场景**（理论）：用 `#### R段（原文引用）` H4 标题替代 `**R — Reading**：` 粗体格式 → R-block 引文长度检查被**完全跳过**——validator 的 R-block 正则 `r'\*\*R — Reading[^\n]*\n\n> (.+?)\n> ——'` 找不到匹配。

**防御**：**所有 R 段都使用 `**R — Reading**：` 粗体行 + H3 标题**——确保两套检测（Section completeness + R-block 长度检查）都覆盖。
